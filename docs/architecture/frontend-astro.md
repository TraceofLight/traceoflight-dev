# Astro Frontend

## Goal

Provide a production-ready frontend baseline for a personal site covering landing, blog, and project pages.

## Related Docs

- Writer/auth implementation details: `docs/architecture/frontend-writer.md`

## Current Scope

- Framework: Astro SSR (`output: server`)
- Adapter: `@astrojs/node` (`mode: standalone`)
- Main routes:
  - `/`
  - `/about`
  - `/blog`
  - `/blog/[...slug]`
  - `/projects`
  - `/projects/[slug]`
  - `/rss.xml`

## Current Structure

- `apps/web/src/layouts`
  - `BaseLayout.astro`: shared page shell (`Header`, `Footer`, `main`, transitions)
  - `BlogPost.astro`: blog detail presentation layout
- `apps/web/src/components`
  - Shell: `BaseHead`, `Header`, `Footer`, `HeaderLink`, `PageTransitions`
  - Reusable cards: `PostCard`, `ProjectCard`
  - Utility: `FormattedDate`
- `apps/web/src/lib`
  - `content-source.ts`: blog content source abstraction (`file` vs `db` provider switch)
  - `projects.ts`: project domain data
  - `page-transition.ts`: reusable transition style generator
- `apps/web/src/styles`
  - `tokens.css`: design tokens and font-face
  - `base.css`: reset/base typography/background
  - `layout.css`: page-level layout utilities and responsive layout
  - `components.css`: shared UI components
  - `global.css`: style entrypoint importing the four modules above

## Content/Data Policy

- Blog content is sourced from the API-backed Postgres store via `lib/blog-db.ts`. Posts are authored in the admin writer (`/admin/posts/new`) and rendered from the DB at request time.
- RSS feeds are per-locale (`/[locale]/rss.xml`) and read from the same DB source, so feed output stays consistent with page rendering. The legacy `/rss.xml` route 302-redirects to `/ko/rss.xml` for existing subscribers.

## UI Composition Policy

- Page-specific files in `src/pages` should compose reusable modules rather than duplicate card markup.
- Shared card rendering is centralized in:
  - `PostCard.astro`
  - `ProjectCard.astro`
- Cross-cutting concerns (SEO metadata, transitions, header/footer) stay in layout/components, not route files.

## SEO and Feed Baseline

- SEO defaults and social metadata are generated by `BaseHead.astro`.
- Canonical URL and per-locale `<link rel="alternate" type="application/rss+xml">` are included by default; the active locale's feed is the primary alternate.
- RSS feed route: `apps/web/src/pages/[locale]/rss.xml.ts` (locale-aware feed with `atom:link rel="self"`, `<language>`, and `content:encoded`).
- Legacy compatibility: `apps/web/src/pages/rss.xml.ts` 302-redirects to `/ko/rss.xml`.

## Runtime and Deployment

- Build and runtime are containerized via multi-stage Dockerfile:
  - `deps` -> `build` -> `runtime`
- Container runtime entry:
  - `node ./dist/server/entry.mjs`
- Compose service:
  - `infra/docker/web/docker-compose.yml`
  - service name: `frontend`
  - internal port: `6543`
  - attached to external edge network: `traceoflight-edge`
- CI entrypoint: `infra/jenkins/Jenkinsfile.frontend` (see `infra/jenkins/README.md`).

## Environment Baseline

- `SITE_URL`: canonical site URL (used in Astro config and metadata)
- `API_BASE_URL`: backend service URL the SSR pages call for posts/series/projects
- `HOST`, `PORT`: runtime bind config for standalone node server
- `INDEXNOW_KEY` (optional): when set, served at `/<key>.txt` for IndexNow ownership verification

## Verification Baseline

Run from `apps/web`:

- `bun run test` (modularization guard tests)
- `bun run build` (Astro production build)
